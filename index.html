🧪 实验一：Web软件开发环境

1.1 实验目的

本次实验旨在搭建一个完整的 Web 软件开发环境，包括安装 Node.js 运行时、npm 包管理器、Git 版本控制工具以及 Hexo 静态博客框架 CLI。通过动手实践，理解 B/S 架构下 Web 开发环境的核心组件构成，掌握从环境安装到项目初始化、本地预览的完整流程。

1.3 实验过程

步骤一：安装 Node.js 与 npm

访问 Node.js 官网 https://nodejs.org/，下载 LTS 版本 的 Windows 安装包。安装时务必勾选 "Automatically install the necessary tools" 并确保 "Add to PATH" 被选中。

安装完成后，打开命令提示符验证：

node -v    # 输出 v22.20.0
npm -v     # 输出 10.x.x

图2-1：验证 Node.js 和 npm 安装成功

步骤二：配置 npm 镜像源（推荐）

为提高国内下载速度，将 npm 默认镜像源切换为 npmmirror：

npm config set registry https://registry.npmmirror.com
npm config get registry    # 验证配置是否生效

图2-2：确认 npm 镜像源已切换

步骤三：安装 Git for Windows

访问 Git 官网 https://git-scm.com/download/win 下载安装包。安装过程中的关键选项：

git config --global user.name "你的姓名"

1.5 知识总结

1. 静态博客 vs 动态博客：

对比维度 静态博客（Hexo） 动态博客（WordPress）
页面生成时机 构建时预先生成所有页面 用户访问时由服务器实时生成
数据库 不需要 通常需要 MySQL
访问速度 极快（纯静态文件） 较慢（需执行后端代码+查数据库）
安全性 高（无后端攻击面） 需额外防护（SQL注入、XSS等）
部署方式 上传静态文件即可 需配置服务器环境和数据库

2. 核心命令速查：

命令 作用
node -v 查看 Node.js 版本
npm install -g <package> 全局安装包
npm install 安装项目依赖
npm config set registry <url> 设置 npm 镜像源
git config --global user.name 设置 Git 用户名
hexo init <folder> 初始化 Hexo 项目
hexo server 启动本地预览服务

1.7 心得体会

本次Web软件开发环境实验，是我第一次完整经历从零搭建开发环境到本地预览的全过程。打开命令行，输入node -v看到版本号跳出的那一刻，一种踏实的成就感油然而生——原来一个可用的开发环境是由这样一个个基础组件支撑起来的。

实验中印象最深的是配置npm镜像源。一开始执行npm install时进度条几乎不动，切换为npmmirror镜像后速度明显提升。这个小小的操作让我体会到：技术学习不仅需要掌握工具本身，更要懂得如何优化工具的使用环境。课程思政常说要“知行合一”，我想，遇到问题、分析原因、动手解决，本身就是知行合一的过程——书本上写“配置镜像源可提高下载速度”只是一句话，真正在命令行里敲下去、亲眼看到速度变化，才算真正理解。

安装Git时配置全局用户信息，git config --global user.name和user.email这两条命令看似简单，却让我意识到：代码从来不是匿名产物，每一行代码背后都对应着一个真实的开发者，都承载着责任。 正如一些课程思政教学案例所强调的，软件开发不仅要掌握技术，更要遵守职业规范、树立责任意识。未来我写的每一行代码，都代表着我的专业素养和职业态度。

通过这次实验，我理解了Web开发环境的核心组件关系——Node.js是运行时，npm是包管理器，Git是版本控制，Hexo是构建工具。每个工具各司其职又协同配合，这种模块化、分层协作的设计思想，也启发我在今后的学习和工作中要学会分工合作、各尽其责。

从命令行中敲下第一行指令，到浏览器里看到Hexo默认首页呈现，这个过程让我真切感受到：技术从来不是冰冷的代码堆砌，而是解决实际问题的工具，是服务他人、服务社会的载体。作为计算机专业的学生，扎实掌握每一项基础技能，既是对自己负责，也是为将来用技术回馈社会打好根基。

2.7 心得体会

安装VS Code前，我一直以为"编辑器不就是输入代码的地方"，打开主界面、看到那一排简洁的活动栏时，我才意识到自己过去对工具的理解有多浅薄。这节实验让我真正开始把VS Code当作一个完整的工作台来认识，而不再只是"打字用的窗口"。

Markdown的写作体验也让我有所感悟：写文档时只需要用#表示标题、用*表示强调，完全不用纠结字号、颜色、行距这些排版问题。这种"关注点分离"的设计哲学——内容创作者专注于内容本身，样式和渲染交给工具处理——对我启发很大。学习生活中，我是否也经常把精力耗费在次要的事情上，反而忽略了真正重要的核心？把注意力聚焦在最有价值的部分，可能是效率提升的关键。

VS Code的插件生态则是另一种触动。我的实验环境里安装的Markdown插件都来自全球各地的开发者，他们素未谋面，却通过开源社区协作，共同打磨出一件件高质量的工具。这让我真切感受到"众人拾柴火焰高"的力量。未来我掌握了一定的开发能力后，也希望参与到这样的开源社区中，为技术生态贡献一份绵薄之力——这不只是技术能力的体现，更是对"责任"二字的践行。

🧪 实验三：Web站点构建

3.1 实验目的

本次实验旨在使用 Hexo 静态站点生成器（SSG） 构建一个完整的个人博客站点，并部署到 GitHub Pages 实现免费上线。通过完整体验 环境准备 → 项目初始化 → 内容编写 → 主题定制 → 本地预览 → 部署上线六个核心环节，理解静态站点生成器的工作原理及其与动态站点的本质区别。

3.2 实验环境

项目 配置信息
操作系统 Windows 11 专业版 22H2
运行时 Node.js v20.x LTS
博客框架 Hexo 7.x
主题 NexT（或其他）
部署平台 GitHub Pages
部署工具 hexo-deployer-git
本地预览 http://localhost:4000

3.3 实验过程

步骤一：创建并编写博客文章

使用 Hexo CLI 创建新文章，编辑 Front-matter 元数据和正文内容：

hexo new "我的第一篇博客"

编辑生成的 Markdown 文件：

---
title: 我的第一篇博客
date: 2026-07-05 10:00:00
tags:
  - 入门
  - Hexo
categories: 技术
---

## 欢迎来到我的博客

这是我的第一篇博客文章。通过 Hexo，我可以用 Markdown 轻松写作。

## 为什么要写技术博客？

- 记录学习和成长
- 分享知识和经验
- 打造个人技术品牌

生成静态页面并预览：

hexo generate    # 或 hexo g
hexo server      # 或 hexo s

图4-1：第一篇博客文章在本地预览中的效果

步骤二：更换主题

以 NexT 主题为例：

git clone https://github.com/theme-next/hexo-theme-next themes/next

修改 _config.yml 中的 theme 字段为 next，然后：

hexo clean && hexo generate && hexo server

图4-2：NexT 主题效果展示

步骤三：创建 GitHub 仓库

在 GitHub 上创建两个仓库：

仓库 名称 可见性 用途
源码仓库 my-blog Private（推荐） 存放 Hexo 源码
部署仓库 [用户名].github.io Public 托管静态页面

⚠️ 关键：部署仓库的名称必须是 你的GitHub用户名.github.io 格式，否则 GitHub Pages 无法正确识别。

图4-3：在 GitHub 上创建部署仓库

步骤四：配置并执行部署

安装部署插件：

npm install hexo-deployer-git --save

在 _config.yml 末尾添加部署配置：

deploy:
  type: git
  repo: https://github.com/你的用户名/你的用户名.github.io.git
  branch: main

执行一键部署：

hexo clean && hexo generate && hexo deploy

图4-4：hexo deploy 部署成功输出

步骤五：访问上线博客

在浏览器中访问 https://你的用户名.github.io，确认博客成功上线。

4.5 知识总结

1. 三种虚拟机网络模式对比：

网络模式 虚拟机↔宿主机 虚拟机↔外网 外网↔虚拟机 虚拟机之间 典型用途
NAT ✅（单向） ✅ ❌ ❌ 日常上网、安装软件
桥接 ✅ ✅ ✅ ✅ 需要被局域网其他设备访问
Host-Only ✅ ❌ ❌ ✅ 搭建隔离的实验私网

2. 双网卡方案的优势：

NAT 网卡保障了虚拟机可以访问互联网（下载软件、更新系统），Host-Only 网卡构建了宿主机与虚拟机之间稳定且隔离的内网环境。两者互补，既不影响上网功能，又保证了多机互联的可靠性。

3. 克隆技术的价值：

通过"模板机 + 完全克隆"的方式，可以将环境搭建时间从 O(n) 降为 O(1)——配置好一台模板机后，克隆几乎零成本复制整个环境，这是云计算和虚拟化技术的核心思想。

  • 避免直接修改 /etc/environment（需要重启，不易版本管理）

1. SSH 密钥认证原理（非对称加密）：

┌──────────────┐                              ┌──────────────┐
│    node1     │                              │    node2     │
│              │   ① ssh-keygen 生成密钥对     │              │
│  id_ed25519  │   ② ssh-copy-id 发送公钥 →→→  │ authorized_  │

│              │   node1 用私钥解密 ←←←        │              │
│              │   验证通过，登录成功           │              │
└──────────────┘                              └──────────────┘

私钥永远不离开本地——这就是"免密"却"更安全"的本质。

断点续传 ❌ 不支持 ✅ 支持
文件过滤 ❌ 不支持 ✅ 支持 --exclude/--include
删除同步 ❌ 不支持 ✅ 支持 --delete
推荐场景 传几个小文件 大目录、频繁同步、自动化部署

  workflow_dispatch:           #   → 也支持手动触发
jobs:                         # 任务定义
  job_id:                     #   任务ID
    runs-on: ubuntu-latest    #   运行环境
    steps:                    #   步骤列表
      - name: 步骤名
        uses: 使用的动作/run: 执行命令

3. Nginx 核心概念：

目录/命令 作用
/etc/nginx/sites-available/ 可用站点配置存放处
/etc/nginx/sites-enabled/ 已启用站点配置（符号链接）
nginx -t 测试配置文件语法
systemctl reload nginx 热重载配置（不中断服务）
root 指令 指定站点的文件根目录
try_files 指令 URL 路由规则
